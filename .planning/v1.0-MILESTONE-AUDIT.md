## Critical Integration Breaks

### BREAK-01: check subcommand exits before recording progress (Critical)

**Requirements:** DRIL-03, PROG-01
**Root cause:** `set -euo pipefail` at `bin/ckad-drill:2` causes the script to exit when `validator_run_checks` returns 1 (any check failed). `validator_exit=$?` on the next line is never reached. `progress_record` is never called for failed drill attempts.

**Impact:** Failed drill sessions are silently lost — users never see their failure history, domain pass rates are inflated (only successes are recorded), and streak tracking is incomplete.

**Fix:** Guard the validator call: `validator_run_checks ... || validator_exit=$?` with `validator_exit=0` initialized before.

### BREAK-02: next/skip namespace cleanup is a silent no-op (Critical)

**Requirements:** DRIL-06, DRIL-07
**Root cause:** `session_require` → `session_read` sets `SESSION_NAMESPACE`. `scenario_cleanup` reads `SCENARIO_NAMESPACE` — a different variable never set in the next/skip paths. The cleanup guard returns early silently.

**Impact:** Every `next` and `skip` command leaves the scenario namespace running in the cluster, accumulating leaked resources.

**Fix:** Set `SCENARIO_NAMESPACE="${SESSION_NAMESPACE}"` before calling `scenario_cleanup` in both subcommands, or modify `scenario_cleanup` to accept an argument.

### BREAK-03: SIGINT trap namespace cleanup broken (Critical)

**Requirements:** DRIL-11
**Root cause:** Same `SESSION_NAMESPACE` vs `SCENARIO_NAMESPACE` variable mismatch in `_drill_cleanup` trap handler.

**Impact:** Ctrl+C during a drill session leaks the namespace.

**Fix:** Same as BREAK-02 — bridge the variable names in the cleanup handler.

## Non-Critical Defects

### DEFECT-01: solution subcommand garbles multi-line steps (Medium)

**Requirement:** DRIL-05
**Root cause:** Uses `while IFS= read -r step; done < <(yq -r '.solution.steps[]')` — the same pattern fixed in Plan 06 for `_validate_single_scenario` was not backported to the solution display path.

**Impact:** Multi-line heredoc solution steps display with each line numbered separately.

**Fix:** Apply the same index-based `yq -r ".solution.steps[${i}]"` pattern from Plan 06.
