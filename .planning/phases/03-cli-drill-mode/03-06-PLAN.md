From bin/ckad-drill _validate_single_scenario (current broken code, lines 224-228):
  while IFS= read -r step; do
    if [[ -n "${step}" ]]; then
      eval "${step}" 2>/dev/null || true
    fi
  done < <(yq -r '.solution.steps[]' "${file}" 2>/dev/null)

Problem: while-read splits at newlines; multi-line heredoc step arrives as many 1-line eval calls.
  Step 2 of sc-multi-container-pod.yaml is a block scalar spanning 15 lines including <<EOF/EOF.
  eval("kubectl apply -f - <<EOF") → incomplete heredoc → silent failure.

Fix strategy: use a C-style for loop with index-based yq extraction:
  local step_count
  step_count=$(yq -r '.solution.steps | length' "${file}" 2>/dev/null || echo "0")
  local i
  for (( i = 0; i < step_count; i++ )); do
    local step
    step=$(yq -r ".solution.steps[${i}]" "${file}" 2>/dev/null)
    if [[ -n "${step}" ]]; then
      if ! eval "${step}"; then
        warn "Solution step $((i+1)) failed — continuing"
      fi
    fi
  done

Key: yq -r ".solution.steps[N]" returns the FULL multi-line string for that index.
     eval receives the complete string including embedded newlines — heredoc works correctly.

Error visibility: replace `2>/dev/null || true` with `if ! eval "${step}"; then warn "..."; fi`
  Still continues on failure (|| true semantics) but prints a warning instead of swallowing silently.
  Keep 2>/dev/null on yq calls only (not on eval) so kubectl errors are visible.

Existing pattern decisions from STATE.md:
  - yq v3 syntax: yq -r '.field // empty' file
  - ((n++)) || true required with set -e for arithmetic from 0
  - eval used in validator for complex commands — same applies here
