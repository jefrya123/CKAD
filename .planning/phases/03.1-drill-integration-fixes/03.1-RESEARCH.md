## Bug Inventory

### BREAK-01 — check subcommand exits silently on validation failure

**Requirements affected:** DRIL-03, PROG-01
**File:** `bin/ckad-drill`, `check)` case arm (lines 302–321)
**Root cause confirmed:**

```bash
# Line 1 of bin/ckad-drill:
set -euo pipefail

# check) case arm:
validator_run_checks "${SESSION_SCENARIO_FILE}" "${SESSION_NAMESPACE}"
validator_exit=$?   # ← NEVER REACHED when validator returns 1
```

`validator_run_checks` returns 1 when any check fails. Under `set -euo pipefail`, that return-1 causes the script to exit immediately. `validator_exit=$?` is dead code in the failure path. `progress_record` is never called for failed drill attempts.

**Fix (confirmed pattern — already used in `_validate_single_scenario`):**

```bash
validator_exit=0
validator_run_checks "${SESSION_SCENARIO_FILE}" "${SESSION_NAMESPACE}" || validator_exit=$?
```

The `|| validator_exit=$?` form is the standard bash idiom for capturing non-zero exits under `set -e`. It is already used in this codebase in the validate-scenario path (Plan 06 established this). The `|| true` form also works but discards the exit code — the `|| validator_exit=$?` form is preferred here because the code below uses `validator_exit` to determine `passed`.

**Side effect to verify:** After the fix, `ckad-drill check` must still exit 0 even when validations fail (the user ran a check, not a command that "failed"). The current code does `exit` on error — the fixed code should not exit non-zero just because checks failed.

### BREAK-02 — next and skip leave namespace running

**Requirements affected:** DRIL-06, DRIL-07
**File:** `bin/ckad-drill`, `next)` and `skip)` case arms (lines 348–363)
**Root cause confirmed:**

```bash
# session_read sets:
SESSION_NAMESPACE=$(jq -r '.namespace // empty' "${CKAD_SESSION_FILE}")

# scenario_cleanup reads:
scenario_cleanup() {
  local namespace="${SCENARIO_NAMESPACE:-}"   # ← different variable
  if [[ -z "${namespace}" ]]; then
    return 0   # ← silently exits, no cleanup
  fi
  ...
}

# next) and skip) case arms call:
session_require     # sets SESSION_NAMESPACE
scenario_cleanup    # reads SCENARIO_NAMESPACE — never set — returns 0 silently
```

`SESSION_NAMESPACE` (from `session_read`) and `SCENARIO_NAMESPACE` (set by `scenario_load`/`scenario_setup`) are two different global variables. In the `next` and `skip` paths, `session_require` is called (sets `SESSION_NAMESPACE`) but `scenario_load` is never called (so `SCENARIO_NAMESPACE` is unset). `scenario_cleanup` reads the empty `SCENARIO_NAMESPACE` and returns early without deleting anything.

**Fix option A — bridge the variable in the caller (minimal change, preferred):**

```bash
next)
  session_require
  SCENARIO_NAMESPACE="${SESSION_NAMESPACE}"
  scenario_cleanup
  session_clear
  ...
```

**Fix option B — teach `scenario_cleanup` to accept an argument:**

```bash
scenario_cleanup() {
  local namespace="${1:-${SCENARIO_NAMESPACE:-}}"
  ...
}
```

Option A is preferred: it changes fewer files (lib/scenario.sh is not modified), and is clearer about intent at the call site. Option B would require updating all other `scenario_cleanup` call sites and updating scenario.sh.

**Same fix applies to `_drill_cleanup` trap handler (BREAK-03 shares this root cause).**

### BREAK-03 — Ctrl+C trap handler leaks namespace

**Requirements affected:** DRIL-11
**File:** `bin/ckad-drill`, `_drill_cleanup()` (lines 29–38)
**Root cause confirmed:**

```bash
_drill_cleanup() {
  trap - INT TERM EXIT
  if [[ -f "${CKAD_SESSION_FILE}" ]]; then
    session_read 2>/dev/null && scenario_cleanup   # ← same mismatch
    session_clear
  elif [[ -n "${SCENARIO_NAMESPACE:-}" ]]; then
    kubectl delete namespace "${SCENARIO_NAMESPACE}" ...
  fi
}
```

`session_read` sets `SESSION_NAMESPACE`. Then `scenario_cleanup` reads `SCENARIO_NAMESPACE`. Same variable mismatch as BREAK-02.

The `elif` branch handles the partial-setup case (where `SCENARIO_NAMESPACE` IS set because `scenario_setup` was interrupted mid-run). This path works correctly. The bug is only in the `if` branch where session.json exists (normal Ctrl+C during active session).

**Fix (same pattern as BREAK-02):**

```bash
_drill_cleanup() {
  trap - INT TERM EXIT
  if [[ -f "${CKAD_SESSION_FILE}" ]]; then
    session_read 2>/dev/null
    SCENARIO_NAMESPACE="${SESSION_NAMESPACE:-}"
    scenario_cleanup
    session_clear
  elif [[ -n "${SCENARIO_NAMESPACE:-}" ]]; then
    kubectl delete namespace "${SCENARIO_NAMESPACE}" --ignore-not-found=true 2>/dev/null || true
  fi
}
```

Note: `session_read 2>/dev/null && scenario_cleanup` was chained with `&&` — if `session_read` returns non-zero (race condition: file deleted between the `-f` check and the `session_read` call), `scenario_cleanup` was skipped. The fix should separate these on individual lines with fallback-safe variable bridging.

### DEFECT-01 — solution subcommand garbles multi-line steps

**Requirements affected:** DRIL-05
**File:** `bin/ckad-drill`, `solution)` case arm (lines 330–337)
**Root cause confirmed:**

```bash
solution)
  session_require
  header "Solution"
  local_step=1
  while IFS= read -r step; do
    printf '%d. %s\n' "${local_step}" "${step}"
    (( local_step++ )) || true
  done < <(yq -r '.solution.steps[]' "${SESSION_SCENARIO_FILE}" 2>/dev/null)
  ;;
```

`yq -r '.solution.steps[]'` outputs each array element as text, but multi-line block scalars are output with embedded newlines. The `while IFS= read -r step` loop reads one newline at a time. Each line of a multi-line step gets its own number. A 3-line heredoc step becomes items 1, 2, 3 instead of item 1.

**Fix (same index-based pattern as `_validate_single_scenario`):**

```bash
solution)
  session_require
  header "Solution"
  local step_count step_i step
  step_count=$(yq -r '.solution.steps | length' "${SESSION_SCENARIO_FILE}" 2>/dev/null || echo "0")
  for (( step_i = 0; step_i < step_count; step_i++ )); do
    step=$(yq -r ".solution.steps[${step_i}]" "${SESSION_SCENARIO_FILE}" 2>/dev/null)
    printf '%d. %s\n' "$(( step_i + 1 ))" "${step}"
  done
  ;;
```

This pattern was already applied and validated in Plan 06 (`_validate_single_scenario`). The fix here is a direct backport of the same pattern to the `solution` display path.

**Shellcheck note:** `local_step=1` at case-arm scope (not inside a function) produces a shellcheck warning — `local` keyword requires being inside a function. The rewrite eliminates this by using a C-style for loop with `step_i` index instead of a while-read + counter.

## Variable Namespace Map

Understanding which globals each component uses is essential to not introduce new bugs:

| Variable | Set by | Read by | Notes |
|----------|--------|---------|-------|
| `SCENARIO_NAMESPACE` | `scenario_load`, `scenario_setup` | `scenario_cleanup`, `_drill_cleanup` elif branch | Set during drill start and validate-scenario |
| `SESSION_NAMESPACE` | `session_read` | `_drill_cleanup` if branch (buggy), `check)` | Read from session.json |
| `SESSION_SCENARIO_FILE` | `session_read` | `check)`, `hint)`, `solution)`, `current)`, `env)` | Path to scenario YAML |
| `SESSION_SCENARIO_ID` | `session_read` | `check)` | Used in `progress_record` |
| `SESSION_END_AT` | `session_read` | `check)` elapsed calc, `env)`, `timer)` | Epoch timestamp |
| `SESSION_TIME_LIMIT` | `session_read` | `check)` elapsed calc | Seconds |
| `SESSION_MODE` | `session_read` | `check)` mode guard | "drill" or "exam" |

**Key insight:** In `next)`, `skip)`, and `_drill_cleanup` (if branch), the only session state available after `session_require`/`session_read` is the `SESSION_*` family. The `SCENARIO_*` family is only set if `scenario_load` or `scenario_setup` has been called in the same process invocation. They are not persisted to session.json, so they are always unset in the next/skip/cleanup paths.

## Test Strategy

### Existing tests (must keep passing)

All 15 tests in `test/unit/drill.bats` currently pass. None of them test the buggy code paths directly — they test session-missing error cases, not active-session behavior. The fixes must not break any existing tests.

### New tests needed

The bugs are in paths that require an active session (a valid `session.json`). All new tests can be unit-style (no cluster required) by creating mock session.json files and mocking `validator_run_checks`, `scenario_cleanup`, and `kubectl`.


**BREAK-02 test: next calls scenario_cleanup with SESSION_NAMESPACE**

This is harder to test in unit style without a cluster (scenario_cleanup calls kubectl). The most practical approach:
- Mock `kubectl` to succeed silently
- Create a session.json with a known namespace
- Mock `scenario_discover` and `_drill_start` to avoid the full drill flow
- Assert `kubectl delete namespace` was called with the correct namespace

Alternatively, test at the library level: verify `SCENARIO_NAMESPACE` is set to `SESSION_NAMESPACE` before `scenario_cleanup` is called, using a mock `scenario_cleanup` that captures its environment.

**BREAK-03 test: _drill_cleanup deletes namespace from session**

Same approach as BREAK-02 but triggering via `kill -INT` or directly calling `_drill_cleanup` with a mock session.json.

**DEFECT-01 test: solution displays multi-line steps as complete numbered steps**


The `solution` subcommand only reads from the YAML file — it does not call kubectl. This makes it straightforward to test with a mock session.json pointing to a temp YAML file.

## Pitfalls

### Pitfall 1: Over-bridging BREAK-01 exit behavior

After the fix, `ckad-drill check` must still exit 0 even when validations fail. The user ran `check` — that command succeeded (it ran validations and recorded results). Exiting non-zero would suggest a tool error, not a validation failure. Confirm that after the fix the exit code from the `check)` case arm is 0.

**Verification:** `ckad-drill check` with failing validations should output the fail results, record to progress.json, and exit 0.

### Pitfall 2: `session_read && scenario_cleanup` chaining in `_drill_cleanup`

The current `_drill_cleanup` uses `session_read 2>/dev/null && scenario_cleanup`. If `session_read` returns non-zero (race: file disappeared between the `-f` check and the read), the `&&` chain skips cleanup entirely. The fix should ensure cleanup happens even if `session_read` returns an error.

```bash
# Safe pattern:
session_read 2>/dev/null || true
SCENARIO_NAMESPACE="${SESSION_NAMESPACE:-}"
scenario_cleanup
```

### Pitfall 3: `local` keyword outside functions (DEFECT-01 fix)

The current `solution)` case uses `local_step=1` which looks like a `local` declaration but is actually a regular global variable assignment. The fix uses `local step_count step_i step` — but `local` is only valid inside functions. Since the `solution)` case arm is at case scope (not inside a function), `local` keyword will be invalid and shellcheck will flag it.

The solution: move the `solution)` logic into a `_cmd_solution` helper function (consistent with `_cmd_status` and `_cmd_validate_scenario` which are already helper functions), OR declare variables without `local` and name them distinctively.

Looking at the existing pattern in the codebase: `_cmd_status` and `_cmd_validate_scenario` are helpers. `check)`, `hint)`, `solution)`, `current)`, `next)`, `skip)` are inline case arms. The simpler fix is to just not use `local` in the inline case arm — use distinct variable names that won't shadow anything. The STATE.md note confirms: "Delegate case arms to _cmd_* helpers: local keyword cannot be used at top-level case scope in bash."

**Resolution:** Either extract into `_cmd_solution()` helper (consistent with existing pattern) or avoid `local` by using non-conflicting variable names at case scope.

### Pitfall 4: `next)` calls `_drill_start` which installs a new EXIT trap

After `scenario_cleanup` and `session_clear`, `next)` calls `_drill_start ""  ""  ""`. `_drill_start` installs a new `trap '_drill_cleanup' INT TERM EXIT` before `scenario_setup`. This is correct — the new trap protects the new scenario setup. But the cleanup for the OLD scenario must complete BEFORE the new trap is installed, or `_drill_cleanup` might try to clean up an already-deleted namespace.

Looking at the code: `next)` calls `scenario_cleanup` then `session_clear` then `_drill_start`. `_drill_start` installs the trap as its last step before `scenario_setup`. This sequence is safe — the old cleanup is done before the new trap is registered.

### Pitfall 5: Progress record domain field

In the `check)` fix, `progress_record` needs the domain as an integer (jq uses it as `$domain` with `--argjson domain`). The current code reads it with:
```bash
domain=$(yq -r '.domain // empty' "${SESSION_SCENARIO_FILE}")
```
This returns a string like "1". `progress_record` passes it as `--argjson domain "${domain}"` which requires valid JSON. A bare integer "1" is valid JSON, so this is fine. But if the YAML has `domain: "1"` (quoted), yq returns the string with quotes, which would need different handling. Looking at scenario fixtures — domains are unquoted integers in YAML, so `yq -r` returns "1", "2", etc. as bare strings that are valid jq integer input.
