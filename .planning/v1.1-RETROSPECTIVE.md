## What We Built

ckad-drill: a pure-bash, terminal-based CKAD exam trainer with real cluster validation. From zero to fully functional in under 3 weeks.

**Deliverables:**
- 8 core libraries (~2000 LOC bash)
- 70 YAML scenarios across 5 CKAD domains (40 drill, 13 debug, 17 learn)
- 3 modes: drill, exam (domain-weighted, timed), learn (concept text + guided exercise)
- 10 typed validation checks (kubectl + jsonpath against live kind cluster)
- 207+ bats unit/integration tests, all passing
- CI pipeline (shellcheck + unit on PRs, integration on main)
- One-liner installer + dev-setup script
- README + CONTRIBUTING with scenario authoring guide

---

## What Went Well

### 1. Pure bash was the right call
The pivot from Go+Bubble Tea to pure bash eliminated build complexity, matched the exam environment, and made validation logic trivial (just kubectl + jq/yq). No compile step, no binary distribution headaches.

### 2. YAML scenario format scales
The typed validation schema (resource_exists, resource_field, container_count, etc.) made 70 scenarios achievable without custom logic per scenario. Adding new scenarios is now just YAML authoring.

### 3. Phased execution worked
8 phases, 24 plans, each building on the last. Phase boundaries caught integration issues early (Phase 3.1 gap closure, Phase 6 validation type fixes). The planning overhead paid for itself.

### 4. Test coverage prevented regressions
207+ tests meant Phase 3.1 integration fixes and Phase 6 content migration could be done confidently. Shellcheck on every file caught bash pitfalls before they became bugs.

### 5. Audit caught real bugs
The milestone audit found the setup.manifest gap (14 broken debug scenarios) that would have been a showstopper. Systematic verification > manual spot-checking.

---

## What Didn't Go Well

### 1. setup.manifest bug shipped across 6 phases
The manifest-only setup path was missing from Phase 2 onward. Phases 3-5 never exercised debug scenarios, so the bug was invisible until Phase 6 content migration. **Lesson:** integration tests for each scenario type should have been written in Phase 2, not deferred to Phase 7.

### 2. Placeholder URLs survived to the finish line
`USER/ckad-drill` and `your-username/ckad-drill` were in 4 files across install.sh, README, and CONTRIBUTING. These should have been parameterized or caught by a CI check. **Lesson:** add a grep-based CI check for placeholder strings.

### 3. dev-setup.sh scope gap
CONTRIBUTING.md claimed dev-setup.sh installed jq/yq/kind, but it only installed bats + shellcheck. Documentation and code diverged because the doc was written before the script. **Lesson:** write docs after implementation, or validate docs against scripts in CI.

### 4. Phase 6 verification not re-run after fixes
06-06 fixed validation type issues but the VERIFICATION.md was never updated. Stale verification docs create false confidence. **Lesson:** re-run verification after every gap-closure plan.

### 5. No live cluster testing in CI
Phase 1 verification is still `human_needed` — cluster lifecycle tests require Docker. The integration job creates a kind cluster but Phase 1's own verification was never automated. **Lesson:** if it's not in CI, it's not verified.
